# jannahclinic — P5a: Notification System — Design

> **Status:** DRAFT — pending user review
> **Date:** 2026-05-20
> **Scope of P5 split:** Original P0 roadmap row `P5 — الإشعارات والبوابة` bundles 5 features (notification center, customer home dashboard, settings, support pages, beauty tips). Decomposed into two sub-projects:
>
> - **P5a (this spec):** Event-driven in-app notification system.
> - **P5b (later spec):** Customer-facing dashboard polish (Home redesign, Settings, Support, Beauty Tips).
>
> **Builds on:** P1 (Appointment + AppointmentTransitionService), P2 (Payment + PaymentService), P3 (MedicalEntry + Prescription + MedicalEntryService / PrescriptionService), HandleInertiaRequests global share.
> **Out of P5a:** SMS/Email channels; push notifications; time-based reminders (24h-before-appointment, etc.); admin-broadcast announcements; per-category opt-out settings.

---

## 1. Motivation & decisions locked

The clinic operates over Inertia + Vue with no realtime layer. Today, every state-changing event in P1/P2/P3 (booking confirmed, receipt approved/rejected, medical entry added, prescription added, …) is invisible to the recipient until they re-fetch the affected page. Customers in particular have no way to see "your receipt was approved" without revisiting the payment page.

Three decisions lock the shape of this work:

1. **In-app only.** No SMS, no email. Laravel `MAIL_MAILER=log` and no SMS provider is configured. Adding either channel transmits PHI/PII off-platform and is an ADR-level decision (ADR-003 mandates encryption at rest; analogous in-transit protections for off-platform delivery need a separate ADR). The notification model is built so a future ADR can flip on a `MailChannel` / `SmsChannel` without schema change.
2. **Event-driven, not time-driven.** Notifications are generated *inside* the same `DB::transaction` as the entity change that caused them, by the existing domain services (PaymentService, AppointmentTransitionService, MedicalEntryService, PrescriptionService, BookingService). Scheduled reminders ("appointment in 24h") are explicitly deferred — they require an idempotency strategy (`last_reminder_at` column / dedup query) that's not needed for event-driven cases.
3. **Append-only, recipient-scoped.** A notification belongs to one user (the `notifiable`). It is never edited or deleted by the application. Only `read_at` flips from null → timestamp. The recipient is the only role that can read or mark-read their own notifications.

## 2. Domain model

### 2.1 `notifications` (Laravel's standard table — used as-is)

Laravel ships a canonical `notifications` table generated by `php artisan notifications:table`. We use it directly with **no extension columns** — the `data` JSON payload carries everything the UI needs.

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid | Laravel default |
| `type` | varchar | FQCN of the Notification class — used as the **category** for filtering |
| `notifiable_type` | varchar | always `App\Models\User` |
| `notifiable_id` | bigint | recipient user id |
| `data` | json | payload (see §3) |
| `read_at` | timestamp nullable | flips null → now() on mark-read |
| `created_at`, `updated_at` | timestamps | |

Indexes (added by migration on top of Laravel's default):
- `(notifiable_type, notifiable_id, read_at)` — composite for the "my unread" query
- `(notifiable_type, notifiable_id, created_at DESC)` — for the notification-center feed

### 2.2 Notification category enum

To give the UI a stable filter set independent of class FQCN, the `data` payload always includes a `category` field. Values, as a PHP enum `App\Enums\NotificationCategory`:

| Value | Generated for |
|-------|---------------|
| `appointment` | Appointment state changes (Confirmed / Cancelled / Rejected / Completed / Rescheduled) |
| `payment` | Payment state changes (Approved / Rejected / Refunded) |
| `medical` | Medical entry created, prescription created |
| `system` | Catch-all (placeholder; no generators in P5a — reserved for P5b) |

UI filters operate on `data->>'category'`. The enum is the source of truth — adding a new category requires editing the enum and the filter chip set together.

### 2.3 Payload shape

Every notification's `data` is a JSON object with this shape:

```json
{
  "category": "appointment",
  "title": "تمّ تأكيد موعدك",
  "body": "موعدك مع د. سارة يوم الأحد 24 مايو الساعة 10:00 تمّ تأكيده.",
  "action_url": "/portal/appointments/123",
  "subject_type": "App\\Models\\Appointment",
  "subject_id": 123
}
```

- `category` — see §2.2.
- `title`, `body` — Arabic, formatted at generation time. No re-rendering at read time.
- `action_url` — where the bell-badge click sends the user. Always a relative URL on the same surface (admin → `/admin/...`, portal → `/portal/...`).
- `subject_type`, `subject_id` — the entity the notification is about. Used by the UI to filter ("show me all notifications about this appointment") and by tests to assert linkage without parsing the URL.

No PHI lives in `data`. Titles/bodies use only the customer's *name* and the appointment date/doctor name — never diagnosis, prescription content, staff_notes, etc. This keeps the notification table out of the ADR-003 encryption surface. If a future change needs to leak PHI into a notification body (e.g., "your prescription for X is ready"), the notification table must move under ADR-003's encryption cast at the column level — that is an ADR change, not a code change.

## 3. Event catalog

Every entry below specifies: which service method generates the notification, who receives it, what the title/body/action_url look like.

| Trigger (service method) | Recipient | category | title | action_url |
|---|---|---|---|---|
| `BookingService::book()` (status=Requested) | Manager(s) + Receptionist(s) | `appointment` | «طلب حجز جديد» | `/admin/appointments/{id}` |
| `AppointmentTransitionService::confirm()` | Customer (appt.customer) | `appointment` | «تمّ تأكيد موعدك» | `/portal/appointments/{id}` |
| `AppointmentTransitionService::reject()` | Customer | `appointment` | «نأسف، تعذّر تأكيد موعدك» | `/portal/appointments/{id}` |
| `AppointmentTransitionService::cancelByCustomer()` | Manager(s) + assigned Doctor | `appointment` | «إلغاء موعد من العميل» | `/admin/appointments/{id}` |
| `AppointmentTransitionService::cancelByStaff()` | Customer | `appointment` | «تمّ إلغاء موعدك» | `/portal/appointments/{id}` |
| `AppointmentTransitionService::reschedule()` | the *other* side (if staff initiated → customer; if customer initiated → manager) | `appointment` | «إعادة جدولة موعد» | depends on recipient surface |
| `AppointmentTransitionService::complete()` | Customer | `appointment` | «اكتمل موعدك — يمكنك تقييم الزيارة لاحقًا» | `/portal/appointments/{id}` |
| `PaymentService::verify()` (Submitted → Paid) | Customer | `payment` | «تمّ تأكيد دفعتك» | `/portal/payments/{id}` |
| `PaymentService::reject()` (Submitted → Rejected) | Customer | `payment` | «الإيصال مرفوض — أعد الرفع» | `/portal/payments/{id}` |
| `PaymentService::markRefunded()` (RefundPending → Refunded) | Customer | `payment` | «تمّ تنفيذ الاسترداد» | `/portal/payments/{id}` |
| `PaymentService::uploadReceipt()` (Pending/Rejected → Submitted) | Manager(s) | `payment` | «إيصال جديد بانتظار المراجعة» | `/admin/payments/{id}` |
| `MedicalEntryService::create()` | Customer | `medical` | «أضاف الطبيب ملاحظة على زيارتك» | `/portal/medical-record/entries/{id}` |
| `PrescriptionService::add()` | Customer | `medical` | «تمّ إضافة وصفة جديدة» | `/portal/medical-record/entries/{entryId}` |

Notifications are NOT generated for:
- `medical-entry updated` / `prescription updated` — too chatty; the original creation event is enough.
- `appointment status no-op transitions` — service raises `InvalidAppointmentTransitionException`, no notification.
- View events (audit log already captures these — different concern).

**Multi-recipient fan-out:** for triggers with multiple recipients (e.g., "new booking → all managers"), the NotificationService loops over the role-scoped query (`User::where('role', UserRole::Manager)->where('is_active', true)`) and dispatches one notification per user inside the same transaction. There is no group/team notifiable — Laravel's morphs are per-user only, and that's fine for a single-clinic install with O(few) managers.

## 4. NotificationService

Single entry point. Mirrors the `AuditLogger` pattern from P3: explicit, transactional, called inline by the originating service.

```php
// app/Domain/Notification/Services/NotificationService.php
final class NotificationService
{
    public function appointmentConfirmed(Appointment $appt): void { /* ... */ }
    public function appointmentCancelledByStaff(Appointment $appt): void { /* ... */ }
    public function appointmentRescheduled(Appointment $appt, UserRole $initiatorRole): void { /* ... */ }
    // ... one method per row in §3 ...

    public function markAsRead(DatabaseNotification $n, User $user): void
    {
        if ($n->notifiable_id !== $user->id) {
            throw new AuthorizationException();
        }
        if ($n->read_at !== null) { return; }
        $n->markAsRead();
    }

    public function markAllAsRead(User $user): int
    {
        return $user->unreadNotifications()->update(['read_at' => now()]);
    }
}
```

Each generator method:
1. Resolves the recipient(s) via an explicit query.
2. Builds the payload dict (see §2.3) with hard-coded Arabic strings.
3. Calls `$user->notify(new DomainNotification($payload))` — one tiny `Illuminate\Notifications\Notification` subclass per category that targets only the `database` channel and returns the payload from `toDatabase()`.

The service is the **only** place that constructs notifications. Controllers, jobs, models, and Vue components never call `notify()` directly.

**Inside-transaction invariant:** every generator method MUST be called inside the same `DB::transaction` that wraps the underlying entity change. If the transaction rolls back, the notification rows roll back with it. This is enforced by convention in the originating services (PaymentService, AppointmentTransitionService, …); each service's method that triggers a notification is already wrapped in `DB::transaction` and the `$this->notifications->...()` call goes inside that closure.

## 5. Service integration points

| Service | Method | New line added |
|---|---|---|
| `BookingService` | `book()` | `$this->notifications->bookingRequested($appt);` |
| `AppointmentTransitionService` | `confirm()`, `reject()`, `cancelByCustomer()`, `cancelByStaff()`, `reschedule()`, `complete()` | one matching call each |
| `PaymentService` | `uploadReceipt()`, `verify()`, `reject()`, `markRefunded()` | one matching call each |
| `MedicalEntryService` | `create()` | `$this->notifications->medicalEntryCreated($entry);` |
| `PrescriptionService` | `add()` | `$this->notifications->prescriptionAdded($prescription);` |

`NotificationService` is constructor-injected. Existing services gain one new constructor arg — already trivial because each is a singleton bound by Laravel's container.

`PaymentService::markRefundPending` (Paid → RefundPending) does **NOT** notify — it's an internal staging state caused by the appointment cancellation cascade, not a customer-visible event. The customer already received the cancellation notification; a second "your payment is pending refund" notification would be noise. `markRefunded` is the visible event.

## 6. UI

### 6.1 Inertia global share

`HandleInertiaRequests::share()` gains a single new prop, lazily evaluated:

```php
'notifications' => fn () => $request->user()
    ? [
        'unread_count' => $request->user()->unreadNotifications()->count(),
      ]
    : null,
```

Only the count ships globally. The full feed lives on a dedicated page and is loaded on visit. This keeps the per-request payload bounded.

### 6.2 Bell badge component

`resources/js/Components/foundation/NotificationBell.vue` — bell icon + numeric badge if `unread_count > 0`. Renders in:
- `AdminShell.vue` — top header, near the user menu.
- `ClientShell.vue` — top header.

Clicking the bell navigates to `/admin/notifications` (staff) or `/portal/notifications` (customer). The bell does NOT open a dropdown preview in P5a — that's polish for P5b.

### 6.3 Notification center page

Two near-identical pages — same component logic, different shell:

- `resources/js/Pages/Admin/Notifications/Index.vue` (under `AdminShell`)
- `resources/js/Pages/Portal/Notifications/Index.vue` (under `ClientShell`)

Layout:
- Page header «الإشعارات» + "تعليم الكل كمقروء" button (only if unread > 0).
- Filter chips: «الكل» / «المواعيد» / «المدفوعات» / «السجل الطبي» / «النظام» — map to the `NotificationCategory` enum. Active chip highlighted.
- Read/unread toggle: «الكل» / «غير المقروءة فقط».
- List: paginated (20/page) via Inertia server-paginated cursor; each row shows title, body (1-line truncate), relative time («منذ ساعتين»), an unread dot, and is clickable → navigates to `action_url` *and* marks the row read in the same request.
- Empty state: «لا إشعارات حتى الآن» with a soft illustration tone (use existing Lucide icon, no new asset).

`AdminDataTable` is **not** used here — the admin Data Table directive [[feedback-admin-data-table]] applies to admin **list/CRUD** pages with row actions, sorting, column visibility, etc. The notification feed is a one-action, one-column read-feed; coercing it into a table adds chrome (column header row, view-options, row selection) that has no UX value for a notification stream. The pages share the AdminShell/ClientShell shell, header, and pagination component conventions but render the feed as a list-of-cards. This is a deliberate exception with rationale logged here so future audits don't flag it.

### 6.4 Mark-read endpoints

| Route | Surface | Action |
|---|---|---|
| `POST /admin/notifications/{id}/read` | Admin | mark one read; redirect to `data.action_url` |
| `POST /admin/notifications/mark-all-read` | Admin | mark all unread for user → redirect to index |
| `POST /portal/notifications/{id}/read` | Portal | mark one read; redirect to `data.action_url` |
| `POST /portal/notifications/mark-all-read` | Portal | mark all read for user → redirect to index |

Routes are named, locked in `tests/Feature/RouteNamesTest.php` (project convention).

### 6.5 Click-to-read race

When the user clicks an unread row, the link both marks-read AND navigates. The handler:
1. Verifies `notifiable_id === auth()->id()` (else 403).
2. Sets `read_at = now()` if null.
3. Redirects to `data.action_url`.

The unread badge is decremented on the next Inertia request — the `notifications.unread_count` prop is re-evaluated. No client-side optimistic update in P5a (keeps the bell badge as a single source of truth).

## 7. Authorization

| Action | Allowed |
|---|---|
| List own notifications | the recipient (any role) |
| Mark own notification read | the recipient |
| List or read someone else's notifications | nobody — 403 |

Enforcement: route-level closure check `abort_unless($notification->notifiable_id === $request->user()->id, 403)`. No policy class — too thin to warrant one. Feature tests assert 403 across the cross-user matrix (customer A reading customer B's notifications, doctor reading manager's, etc.).

Staff surface (`/admin/notifications/*`) is gated by the existing `role:manager,doctor,receptionist` middleware on the admin route group. Customer surface (`/portal/notifications/*`) is gated by the existing `role:customer` middleware on the portal route group. A receptionist cannot read a doctor's medical-category notifications via the admin page because notifications are scoped to `notifiable_id = auth()->id()` — i.e., the receptionist sees only notifications addressed to *them*. The category filter operates on their own feed.

## 8. Testing

Following the project's existing test stratification:

**Unit / service tests (Pest)**
- `NotificationServiceTest` — one `it()` per generator method asserting (a) recipient correctness, (b) payload shape, (c) category value, (d) action_url format, (e) inside-transaction (rollback test: throw mid-transaction → notification row absent).
- Test fixtures use `User::factory()` for each role; no HTTP layer.

**Feature tests (Pest, real PostgreSQL via the existing test DB harness)**
- `NotificationCenterTest` — list/filter/mark-read flows for admin and portal:
  - cross-user 403 matrix (customer A ↔ B, doctor ↔ manager)
  - filter by each category returns only matching rows
  - mark-all-read flips every unread row for the user, leaves other users untouched
  - pagination meta correctness (page 1 of N, next/prev links)
- `BellBadgeShareTest` — the Inertia share prop `notifications.unread_count` returns correct count per user; closure lazy-evaluates (no count query when prop unread).
- `EventToNotificationTest` — full happy paths for each generator: call the originating service method, assert the right user got the right notification with the right payload.

**Vitest**
- `NotificationBell.vue.spec.js` — renders badge when `unread_count > 0`, hides badge when 0, links to the correct surface URL based on shell.

**Manual smoke (golden path)**
- Doctor confirms an appointment → customer logs in → sees badge=1 → opens portal/notifications → sees row → clicks → arrives on the appointment page → returns → badge=0.
- Manager rejects a receipt → customer sees badge → reads → opens payment page → re-uploads → manager sees badge → marks as paid → customer sees badge again.

Target coverage: **+18 Pest tests**, **+1 Vitest spec**.

## 9. Out of scope (deferred)

| Item | Why deferred | Reactivation trigger |
|---|---|---|
| SMS channel | No provider configured; PII off-platform requires ADR | clinic requests + chosen provider + new ADR |
| Email channel | `MAIL_MAILER=log`; same off-platform PII argument | same |
| Time-based reminders (e.g., 24h before appointment) | Requires idempotency strategy + scheduler entry; no operational need surfaced yet | clinic measures no-show rate and asks for reminders |
| Bell-dropdown preview (no page navigation) | UI polish; the dedicated page is sufficient for P5a | P5b polish wave |
| Per-category opt-out user preferences | YAGNI for single-clinic + 4 roles; introduce when a real complaint comes in | user feedback / category-noise complaints |
| Admin broadcast announcements (manager → all customers) | Different recipient query (everyone), different generator surface | feature request |
| Realtime push (Echo/Pusher/WebSocket) | Inertia full-page nav already re-reads `unread_count` per request — sufficient for clinic UX | high-traffic phase or staff insist on instant pop-ups |

Each deferred item is small enough to land as a P5a-extension PR; the schema (Laravel default `notifications` + JSON `data`) carries them all without migration.

## 10. Implementation sequencing (informs the plan)

The plan should follow this order — every step ends green (Pest + Vitest + Pint + PHPStan + Vite build):

1. **Migration + enum + Notification subclasses + NotificationService skeleton + unit tests for the service.** No integration with other services yet.
2. **Routes + controllers + auth checks (admin + portal) + feature tests for mark-read and cross-user 403 matrix.** Stub the index page renders.
3. **Bell component + Inertia share + AdminShell/ClientShell integration + Vitest spec.**
4. **Notification center page (filters + pagination + mark-all + list rows) — admin + portal.** Feature tests for filters, pagination meta, click-through navigation.
5. **PaymentService integration — `verify`, `reject`, `markRefunded`, `uploadReceipt`.** EventToNotificationTest covers each.
6. **AppointmentTransitionService + BookingService integration.** EventToNotificationTest covers each.
7. **MedicalEntryService + PrescriptionService integration.** EventToNotificationTest covers each.
8. **DoD gate + tag `p5a-notifications`.** Full Pest + Vitest + Pint + PHPStan + Vite build + manual golden-path smoke + commit + push + tag.

Each step is independently shippable: after step 1, the service works; after step 2, customers can mark notifications read; after step 3, the bell shows up; after step 4, the center is usable; steps 5-7 wire the actual events.
